Understanding Health Insurance in Jamaica

Exploring the landscape of public and private healthcare coverage.

Jamaica: Population and Insurance Landscape

Jamaica has a population of approximately 2.8 million people. Around 25% of the population is covered by private insurance schemes, with about 5% of this group covered by the National Insurance Scheme of Jamaica.

Private Insurance Trends:

  • Growing Market
  • Comprehensive Coverage, including pre-existing and chronic conditions.
  • Wider range of products and services (e.g., wellness programs).
  • Adoption of digital solutions (online policy management, telemedicine, mobile apps).
  • Driven by economic growth, increased disposable incomes, GDP growth, GDP per capita growth, and Purchasing Power Parity increase.

National Health Insurance Plan (NHIP)

The National Health Insurance Plan (NHIP) is a risk-pooling mechanism designed to offer financial protection against the burden of illness, ensuring access to care for those who need it.

It also serves as a payment mechanism for healthcare providers, offering prompt and regular reimbursements.

Furthermore, the NHIP acts as a valuable database for planners and policymakers, providing insights into disease patterns, service utilization, health expenditure, and patient preferences.

Key Considerations:

  • Mandatory vs Compulsory implementation.
  • Public/Private partner coverage models.
  • Integration of Micro-insurance schemes.

Data Ingestion and Training Process

Phase One: Discovery (One Month)

This phase involves identifying, documenting, and ingesting all viable data sources into the data pipeline. Both digital and non-digital data (NDD) are scraped. NDD requires scanning and processing.

List of Documents for Data Ingestion:

Structured Data:
  • Policy Details
  • Identity Proof
  • Medical Reports
  • Original Claim Form
  • Doctor's Prescription
  • Other Potential Documents (Indoor case papers, Ambulance receipts, Pharmacy bills, FIR, Proof of income/employment, Pension Forms)
  • Internal database data
  • Lab Reports or Test results
  • Additional Considerations for Expats
Unstructured/Semi-structured Data:
  • Emails between patient and Insurance/Healthcare provider
  • PDFs, notes, or handwritten documents
  • Non-digital data (NDD) crucial for context
  • Inter-agency documents
  • List of illnesses with no coverage

Phase Two: Data Ingestion Layer/Training (2-3 Months)

In this phase, the curated data is ingested, and the training process for the Generative AI architecture begins. This involves tokenization, where data is converted into tokens to estimate training time and hardware requirements.

Token Estimation:

Estimates are made for both structured and unstructured data, considering factors like document length, words per page, relevance, and instance rates. Image tokens are significantly larger than text tokens.

Estimated Total Training Tokens Range: 60 billion - 150 billion (including Image Factor)

Based on these estimates, an LLM/Agentic system is engineered to handle the client's requirements.

LLM Query Handling with Access Control:

This process ensures secure and efficient functionality by combining vectorization, access control, and query-time filtering, respecting user roles and privileges when accessing data.

Infrastructure and Storage Requirements

Storage Requirement for Incoming Data

Similar to the token estimation, there would be a need to provide adequate storage for daily incoming unstructured and semi-structured data.

Sizing Analysis for Vector Database Supporting RAG/AGENTS

Key Assumptions:

  • Corpus Size: 150 billion tokens (text to be indexed for retrieval).
  • Embedding Dimension: 1024 (float32, 4 bytes per value).
  • Text Chunk Size: 512 tokens per chunk (common default).
  • Token Storage: 4 bytes per token (UTF-8 or similar encoding).
  • Indexing Method: Hybrid in-memory/disk (e.g., DiskANN) for balance of speed and cost.

1. Vector Storage

Number of Chunks:

150B tokens / 512 tokens per chunk ≈ 293M chunks

Vector Size per Chunk:

1024 dimensions × 4 bytes = 4 KB

Total Vector Storage:

293M × 4 KB = 1.17 TB

2. Text Storage

Text per Chunk:

512 tokens × 4 bytes = 2 KB

Total Text Storage:

293M × 2 KB = 572 GB

3. Index Storage

Disk-Based Index (e.g., DiskANN):

Roughly equal to vector size (~1.2 TB).

Total Disk Storage (Vectors + Text + Index):

1.17 TB + 0.57 TB + 1.2 TB ≈ 3 TB (with 20% overhead: ~3.6 TB).

4. Memory Requirements

DiskANN (Hybrid SSD/RAM):

Caches ~10–20% of the index in memory.

1.2 TB × 0.2 = 0.24 TB (240 GB RAM)

Recommendation: 256–512 GB RAM for headroom.

Pure In-Memory Index (e.g., HNSW):

1.2 TB RAM, requiring distributed nodes (e.g., 10+ nodes with 128 GB each).

5. Network and Compute

Query Throughput: Scales with nodes (e.g., 1k–10k queries/second with distributed clusters).

CPU/GPU: Modern CPUs (e.g., Intel Xeon) or GPUs (for accelerated similarity search).

6. Design Considerations

  • Chunk Size: Smaller chunks → more vectors (e.g., 256 tokens → ~586M vectors, doubling storage).
  • Quantization: Use 8-bit SQ (reduces vector storage by 75% but impacts accuracy).
  • Distributed vs. Single-Node:
    • Single-Node: Feasible with DiskANN (~512 GB RAM + 4 TB SSD).
    • Distributed: Required for pure in-memory indices (e.g., Milvus cluster).
  • Metadata: Add 10–20% storage for additional metadata (e.g., timestamps, source URLs).

Summary of Memory Requirements

Component Size Notes
Vectors (1024D) 1.17 TB Float32 embeddings.
Text Chunks 572 GB 512 tokens/chunk.
Index (DiskANN) 1.2 TB Hybrid SSD storage.
RAM (DiskANN) 240–512 GB For cached graph nodes.
Total Storage 3.6 TB (with overhead) SSDs recommended for fast retrieval.

Adjust based on query latency/throughput requirements and accuracy trade-offs. The overhead on RAM can be reduced by lowering the precision and/or size of the embedding.
